# OpenClaw 用户会话隔离与权限控制方案

## 概述

本文档记录了 OpenClaw 系统中用户会话隔离和权限控制的讨论内容、现状分析以及可能的解决方案。

## 一、现状分析

### 1.1 当前能力

#### ✅ 已具备的能力

1. **用户识别**
   - 可以通过 `sender_id` 唯一标识不同用户
   - 支持跨平台用户识别（飞书、企业微信、钉钉、Telegram 等）
   - 每个消息渠道都提供类似的用户元数据

2. **会话隔离**
   - 每个对话都有独立的 `chat_id`
   - 不同用户的对话上下文完全隔离
   - 群聊中可以区分每条消息的发送者

3. **用户记忆**
   - 可以针对不同用户存储和回忆不同的信息
   - 支持用户级别的长期记忆管理

### 1.2 各平台用户标识对比

| 信息类型 | 飞书 | 企业微信 | 说明 |
|---------|------|---------|------|
| 用户 ID | `sender_id` | `userid` | 唯一用户标识 |
| 会话 ID | `chat_id` | `chatid` | 唯一会话标识 |
| 用户姓名 | `sender.name` | `name` | 用户显示名称 |

### 1.3 当前局限

#### ❌ 缺少的能力

1. **权限控制**
   - 没有内置的用户权限控制机制
   - 无法基于用户身份限制访问某些 skills
   - 无法基于用户身份限制访问某些 MCP 服务
   - 没有内置的访问控制列表（ACL）

2. **安全悖论**
   - 如果用配置文件控制权限，任何用户都可以修改配置文件
   - 任何用户都可以给自己提升权限
   - 技术上的权限控制形同虚设

## 三、权限控制方案

### 方案1：在 Skill 层面实现（推荐用于简单场景）

#### 实现思路
在具体的 Skill 中添加用户权限检查逻辑。

#### 优点
- 实现简单，不需要修改核心代码
- 灵活性高，可以针对每个 Skill 定制权限逻辑
- 易于理解和维护

#### 缺点
- 权限逻辑分散，难以统一管理
- 每个 Skill 都要重复实现权限检查
- 用户可以要求修改 Skill 代码来绕过权限

#### 代码示例

```javascript
// 伪代码示例
const allowedUsers = [
  'ou_c8a9cafe83d065f5bde17b4365a7701c', // 管理员用户
  'ou_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'    // 其他授权用户
];

async function executeSkill(params, context) {
  const senderId = context.senderId;
  
  // 权限检查
  if (!allowedUsers.includes(senderId)) {
    return {
      success: false,
      message: "抱歉，你没有权限使用这个功能"
    };
  }
  
  // 正常执行功能
  // ...
}
```

### 方案2：使用配置文件（推荐用于中等复杂度场景）

#### 实现思路
维护一个统一的权限配置文件，定义每个用户可以访问的功能。

#### 优点
- 权限逻辑集中管理
- 修改配置不需要改代码
- 可以定义角色和权限组

#### 缺点
- 用户可以要求修改配置文件
- 配置文件需要额外的保护机制

#### 配置文件示例

```json
{
  "version": "1.0",
  "roles": {
    "admin": {
      "description": "管理员，拥有所有权限",
      "permissions": ["*"]
    },
    "user": {
      "description": "普通用户，拥有基本权限",
      "permissions": [
        "skills.daily-report",
        "skills.calendar",
        "mcp.files.read"
      ]
    },
    "guest": {
      "description": "访客，只有只读权限",
      "permissions": [
        "skills.calendar.read"
      ]
    }
  },
  "users": {
    "ou_c8a9cafe83d065f5bde17b4365a7701c": {
      "name": "张三",
      "role": "admin"
    },
    "ou_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx": {
      "name": "李四",
      "role": "user"
    }
  }
}
```

#### 权限检查逻辑

```javascript
function checkPermission(senderId, permission, config) {
  const user = config.users[senderId];
  if (!user) {
    return false; // 用户未定义，默认拒绝
  }
  
  const role = config.roles[user.role];
  if (!role) {
    return false; // 角色未定义，默认拒绝
  }
  
  // 检查权限
  if (role.permissions.includes("*")) {
    return true; // 管理员拥有所有权限
  }
  
  return role.permissions.includes(permission);
}
```

### 方案3：在 OpenClaw 核心层面实现（最佳方案，但需要修改核心）

#### 实现思路
修改 OpenClaw 的核心代码，在工具调用层面添加权限检查。

#### 优点
- 权限控制在核心层面，无法绕过
- 统一的权限管理接口
- 支持细粒度的权限控制

#### 缺点
- 需要修改 OpenClaw 核心代码
- 实现复杂度较高
- 超出当前能力范围

#### 架构设计

```
┌─────────────────┐
│   用户消息       │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  消息通道层      │
└────────┬────────┘
         │
         ▼

└────────┬────────┘

         ▼

└─────────────────┘
```

### 方案4：只读配置 + 重启生效（折中方案）

#### 实现思路
- 配置文件设为只读
- 修改后需要重启 OpenClaw 才能生效
- 即使用户让我修改了配置，也不会立即生效

#### 优点
- 配置文件被修改后不会立即生效
- 有时间窗口发现和回滚未授权的修改
- 实现相对简单

#### 缺点
- 需要手动重启 OpenClaw
- 用户体验不太好
- 还是无法阻止用户修改配置

#### 实现步骤

1. 将配置文件设为只读：
```bash
chmod 444 permissions.json
```

2. 在代码中添加配置版本检查：
```javascript
let configVersion = null;
let config = null;

function loadConfig() {
  const newConfig = JSON.parse(fs.readFileSync('permissions.json'));
  if (configVersion && newConfig.version !== configVersion) {
    console.warn('配置文件已修改，需要重启 OpenClaw 才能生效');
    return config; // 返回旧配置
  }
  configVersion = newConfig.version;
  config = newConfig;
  return config;
}
```

### 方案5：接受现实，用管理流程控制（最实际的方案）

#### 实现思路
- 不通过技术手段限制，而是通过管理流程
- 只给可信的用户访问权限
- 记录所有操作日志

#### 优点
- 实现最简单
- 灵活性最高
- 不会影响用户体验

#### 缺点
- 依赖人工管理
- 没有技术上的强制限制

#### 管理建议

1. **访问控制**
   - 只给可信的员工访问 OpenClaw 的权限
   - 定期审查访问权限

2. **审计日志**
   - 记录所有重要操作
   - 定期审查操作日志

3. **使用协议**
   - 制定 OpenClaw 使用规范
   - 明确禁止的行为

4. **教育培训**
   - 培训用户正确使用 OpenClaw
   - 强调安全意识

## 五、推荐方案

### 阶段1：初期（当前）

**推荐方案5：管理流程控制**

理由：
- 快速上线，不需要开发
- 灵活性最高
- 适用于用户数量少、互相信任的场景

### 阶段2：中期（用户增长）

**推荐方案2 + 方案4：配置文件 + 只读+重启**

理由：
- 可以统一管理权限
- 有一定的安全保障
- 实现难度适中

### 阶段3：长期（生产环境）

**推荐方案3：核心层面实现**

理由：
- 安全性最高
- 无法绕过
- 适合企业级生产环境

## 六、实施建议

### 6.1 立即可以做的

1. **记录用户信息**
   - 为每个用户创建一个简单的用户档案
   - 记录用户的基本信息和使用习惯

2. **操作日志**
   - 开始记录重要的操作
   - 包括用户、时间、操作内容

3. **使用规范**
   - 制定简单的使用规范
   - 明确哪些操作是允许的，哪些是禁止的

### 6.2 下一步可以做的

1. **实现简单的权限配置文件**
   - 参考方案2的配置文件格式
   - 先在内部测试使用

2. **添加操作审计功能**
   - 在关键操作前记录日志
   - 定期审查操作日志

3. **用户培训**
   - 培训用户正确使用 OpenClaw
   - 强调安全意识

## 七、总结

OpenClaw 当前已经具备了良好的用户识别和会话隔离能力，但缺少内置的权限控制机制。在当前阶段，建议通过管理流程来控制访问，同时逐步探索技术方案。长期来看，最好在 OpenClaw 核心层面实现权限控制机制。
