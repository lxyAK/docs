# 文档目录组织规范

## 目录结构

```
docs/
├── analysis/                        # 技术分析文档
│   └── {项目名}/                    #   按项目名分目录
│       └── [主题]分析.md
├── architecture/                    # 架构文档
│   └── {项目名}/
│       └── [主题]方案.md
├── deployment/                      # 部署文档
│   └── {项目名}/
│       └── [主题]部署指南.md
├── guides/                          # 使用指南
│   └── {项目名}/
│       └── [主题]使用指南.md
└── openclaw-source-analysis/        # OpenClaw 源码专题（独立连载系列）
```

## 核心规则

**类型分主目录，项目名分二级目录。**

1. 先确定文档类型 → 选择主目录（analysis/architecture/deployment/guides）
2. 再确定项目名 → 在主目录下创建或使用已有的项目子目录
3. 文件放在对应的项目子目录内

## 各主目录说明

### analysis/ - 技术分析
- 技术方案对比、可行性分析、深度技术分析、架构决策分析、集成方案分析
- **命名规范**：`[主题]分析.md`、`[主题]可行性分析.md`、`[主题]方案对比.md`

### architecture/ - 架构设计
- 系统架构设计、完整技术指南、定制架构方案、实施方案
- **命名规范**：`[主题]架构方案.md`、`[主题]完整指南.md`、`[主题]实施方案.md`

### deployment/ - 部署指南
- 部署指南、发版指南、运维手册
- **命名规范**：`[主题]部署指南.md`、`[主题]发版指南.md`

### guides/ - 使用指南
- 使用说明、操作手册、集成使用指南
- **命名规范**：`[主题]使用指南.md`、`[主题]使用说明.md`

### openclaw-source-analysis/ - 源码专题
- OpenClaw 源码深度分析系列（7 篇连载）
- 有独立的 README.md 索引和阅读路径
- 此目录为独立专题，不按项目名再分目录

## 文件命名规范

- **中文为主**，避免英文和中文混用
- **避免空格**，用连字符替代（如 `供应链攻击检测脚本使用说明.md`）
- **类型后缀**：分析 → `xxx分析.md`，方案 → `xxx方案.md`，指南 → `xxx指南.md`
- **项目名做目录**：文档内容中不需要重复项目名前缀，目录已经表达了归属

## 新建文档流程

1. 确定文档类型 → 选择主目录
2. 确定项目名 → 查看主目录下是否已有该项目子目录
   - 已有 → 直接放入
   - 没有 → 创建新目录
3. 按命名规范起文件名
4. 写文档内容
5. 更新 `README.md` 索引
6. 提交：`git add && git commit -m "docs(项目名): 描述"`

## 源码分析类文档要求

所有源码分析类文档必须在文档开头标注：

```
> **分析项目**：{项目名}
> **仓库地址**：{Git 仓库 HTTPS 地址}
> **分析日期**：{YYYY-MM-DD}
```
